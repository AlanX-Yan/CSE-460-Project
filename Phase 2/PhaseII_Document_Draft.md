## 1. Document Explaining and Justifying Design Patterns

### Design Pattern 1: Facade Pattern

The Facade Pattern provides a simplified interface to a larger and more complex set of classes. In the SunDevil Connect system, many user actions require several internal classes to work together. For example, when a student registers for an event, the system may need to check authentication, retrieve the event, verify that the event is still open, check capacity, create an event registration, and update the event's remaining spots. If every user interface page directly communicates with all of these controllers and entity classes, the design becomes tightly coupled and difficult to maintain. To improve this, the refined design introduces a `SunDevilConnectFacade` class as the main access point between the boundary/UI classes and the internal application logic.

In this design, pages such as `EventSearchPage`, `EventDetailsPage`, `ClubPage`, `ClubDashboardPage`, and `AdminPanelPage` send user requests to the facade instead of directly managing multiple controllers. The facade then coordinates with `AuthController`, `EventController`, `ClubController`, and `AdminController` to complete the requested action. For example, `EventDetailsPage` can call `registerForEvent(studentId, eventId)` on the facade, and the facade will delegate the needed work to the event and registration logic. Similarly, `ClubDashboardPage` can call methods such as `createEvent()` or `postAnnouncement()` without knowing every internal class involved.

This pattern improves the design because it reduces coupling between the user interface and the business logic. If the internal implementation changes later, such as adding a database layer, changing registration rules, or modifying how clubs are approved, the UI pages do not need to be rewritten as long as the facade's public methods remain stable. The facade also makes Phase III implementation more manageable because the user interface can be built around a small set of clear operations. This supports maintainability, readability, and scalability. Since SunDevil Connect includes different roles and many workflows, the Facade Pattern gives the system a cleaner structure and helps prevent the boundary classes from becoming overloaded with business rules.

### Design Pattern 2: State Pattern

The State Pattern allows an object to change its behavior when its internal state changes. This pattern is useful when an object has a lifecycle with different rules in each stage. In SunDevil Connect, the `Event` class is a strong candidate for the State Pattern because an event can move through several conditions during its lifetime. An event may begin as a draft, become published, reach full capacity, or be cancelled. Each of these states should allow different actions. For example, a draft event can be edited and published, but students should not register for it yet. A published event can accept registrations. A full event should reject normal registration or move students to a waitlist if that feature is added later. A cancelled event should not allow new registrations and should notify affected students.

In the refined class diagram, the `Event` class has a reference to an `EventState` interface or abstract class. Concrete state classes include `DraftEventState`, `PublishedEventState`, `FullEventState`, and `CancelledEventState`. Each state class implements behavior such as `publish()`, `registerStudent()`, `cancel()`, and `updateDetails()` according to the rules of that state. Instead of placing all status rules inside a large conditional statement in `EventController`, the controller can call the current state's method through the `Event` object. The state object then decides whether the action is valid and how the event should transition to the next state.

This pattern improves the design by keeping state-specific behavior organized and easy to modify. Without the State Pattern, the system would likely contain many if-else or switch statements checking whether an event is draft, published, full, or cancelled. As the project grows, those conditionals would become harder to test and more likely to contain errors. The State Pattern separates these rules into focused classes, which makes the design more readable and extensible. If the system later adds new states, such as `WaitlistedEventState` or `ArchivedEventState`, developers can add new classes without heavily changing existing controller code. This is especially helpful for SunDevil Connect because event registration and event management are central features. The pattern improves correctness by making invalid transitions easier to prevent, and it improves maintainability by placing each event state rule in the class where it belongs.

### Design Pattern 3: Command Pattern

The Command Pattern encapsulates a request as an object. This allows the system to treat user actions as independent command objects that can be executed, logged, validated, queued, or even undone in the future. In SunDevil Connect, this pattern is useful for actions performed by club leaders and administrators because those actions often change important system data. For example, a club leader may create an event, update an event, approve a member request, or post an announcement. An administrator may approve a new club or review flagged content. These actions should be handled consistently and should be easy to track because they affect students, clubs, and campus activity information.

In the refined design, the system includes a `DashboardCommand` interface with an `execute()` method. Concrete command classes include `CreateEventCommand`, `UpdateEventCommand`, `ApproveMemberCommand`, `PostAnnouncementCommand`, and `ReviewFlaggedContentCommand`. Each command object contains the information needed to complete its action, such as the acting user, the target club or event, and the input data from the dashboard. The `ClubDashboardPage` and `AdminPanelPage` can create or send command requests through the `SunDevilConnectFacade`, and the appropriate controller can execute the command. For example, approving a member request can be represented by an `ApproveMemberCommand` that updates a `ClubMembership` record and may later trigger a notification or audit entry.

This pattern improves the design because it separates the request itself from the object that performs the request. Instead of writing all dashboard behavior directly inside the UI pages or controllers, each action has its own command class with a clear responsibility. This makes the system easier to extend when new dashboard operations are added. For example, if a future version adds "suspend club," "duplicate event," or "restore announcement," each operation can be introduced as a new command without rewriting the existing dashboard structure. The Command Pattern also supports better testing because each command can be tested individually. It also prepares the system for useful future features such as activity logs, undo operations, permission checks, or delayed processing. Since SunDevil Connect has multiple user roles with different management actions, the Command Pattern gives the design a flexible and organized way to handle role-based operations while keeping the controllers from becoming too large.
